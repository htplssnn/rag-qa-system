RAG 知识库问答系统 —— 面试讲解稿
=====================================

（面试官看着你的项目问"这个项目做了什么？"时怎么说）


=== 30 秒电梯演讲版 ===

"我做了一个 RAG 知识库问答系统，简单说就是：上传任意文档建一个私有
知识库，然后用自然语言提问，AI 会根据文档内容回答，而不是瞎编。

最大的特点是：我用 Python 标准库从零实现，没有用 LangChain 框架，
所以对 RAG 每个环节——文档切分、向量检索、Prompt 工程——都有深入理解。"


=== 2 分钟详细版 ===

【背景】
"大语言模型有两个痛点：一是会幻觉，编造不存在的信息；二是知识有
截止日期，不知道最新的或私有的信息。RAG 就是解决这两个问题的方案。"

【系统架构】
"我的系统分两个阶段：
- 建库阶段：文档上传后，先切分成文本块（保证语义完整），再向量化
  存入向量数据库
- 查询阶段：用户提问后，先检索最相关的文本块，再组装成 Prompt 送
  给大模型生成回答

两个阶段通过向量数据库连接，这是 RAG 最核心的设计。"

【技术选型】
"- 语言：Python 标准库 + http.server，零外部依赖就能跑
- 检索：自研的关键词 n-gram 匹配算法，中文效果好
- 大模型：通过 OpenAI 兼容接口调用 DeepSeek
- 前端：纯 HTML/CSS/JS 内嵌在 Python 里

选 Python 标准库而不是 LangChain，是因为我想深入理解每个环节，
而不是当 API 调用侠。"

【效果展示】
"可以现场演示：上传文档 → 提问 → AI 基于文档回答，并标注来源。
对于文档没有的信息，系统会诚实告知，不会瞎编。"


=== 技术亮点（面试加分点）===

1. 模块解耦设计
   "检索和生成是解耦的——换向量数据库不影响生成逻辑，换大模型不影
   响检索逻辑。这在实际工程中很重要。"

2. 降级策略
   "当 Embedding API 不可用时，自动降级为关键词检索，不会直接挂掉。"

3. 中文友好的文本切分
   "我用递归切分，优先在自然分隔符处切——段落、句号、逗号——而不是
   简单按字符数暴力切，这样切出来的块语义更完整。"

4. 来源追溯
   "每条回答都标注信息来源，这在企业场景是刚需——你不能告诉老板
   'AI 说的'，你得说'根据第X份文档'。"


=== 可能的追问 & 回答 ===

追问：词级别的 n-gram 检索和语义检索有什么区别？
答：n-gram 是精确匹配，优点是快、不需要 Embedding API，缺点是
近义词匹配不到；语义检索能理解"开心"和"高兴"是近义的，但需要
Embedding 模型。我这个项目先用 n-gram 快速验证了 RAG 的完整流程，
后续接入 Embedding API 只需改一行配置。

追问：文本切分的 chunk_size 怎么选？
答：我设了 400 字 + 50 字 overlap。太小则语义不完整，太大则检索
精度下降。这是经验值，实际项目中要根据文档类型调参。还有个技巧：
按自然分隔符切分比按固定字符数切的质量好很多。

追问：为什么选 DeepSeek 而不是 ChatGPT？
答：DeepSeek 性价比高，API 兼容 OpenAI 格式所以切换方便，而且
注册送 500 万 token 额度，适合个人项目。代码里改一行 base_url
就能切到通义千问或智谱。

追问：向量检索是怎么实现的？
答：底层用余弦相似度算两个向量的距离。余弦相似度对向量长度不敏感，
只关心方向，是语义检索最常用的度量。相似度最高的 Top-K 个片段
作为上下文送给大模型。

追问：如何评估 RAG 系统的效果？
答：三个维度——
- 检索准确率：检索到的片段是否和问题相关（用人工标注的测试集）
- 回答准确率：回答是否基于检索内容、有没有幻觉
- 端到端响应时间：从提问到得到回答的耗时

追问：这个项目有什么难点？
答：最大的难点是中文文本的切分和检索。中英文不一样——英文天然有
空格分词，中文是一串连续字符。我用了 n-gram 字符级匹配解决了这个
问题。另外在 Python 标准库限制下实现完整系统也锻炼了工程能力。
