=== 第一类：技术理解类（证明 RAG 能准确回答专业问题）===

Q1: 什么是 RAG？
预期回答要点：
- 检索增强生成，结合信息检索和大模型生成
- 四个好处：减少幻觉、知识可更新、来源可追溯、领域可定制
- 两个阶段：离线建库 + 在线查询

Q2: RAG 的核心流程分为哪几个步骤？
预期回答要点：
- 离线：文档加载 → 文本切分 → 向量化 → 向量存储
- 在线：问题向量化 → 相似检索 → Prompt 组装 → LLM 生成

Q3: RAG 系统中文本切分为什么需要 overlap？
预期回答要点：
- 防止关键信息被切断在 chunk 边界
- 通常保留 50 字左右的重叠

Q4: RAG 有哪些进阶方向？
预期回答要点：
- Agent 化、多模态 RAG、Graph RAG、Self-RAG

Q5: 常见的向量数据库有哪些？各有什么特点？
预期回答要点：
- ChromaDB：轻量级、Python 原生
- FAISS：检索快、C++ 实现
- Milvus：分布式、适合生产

=== 面试官可能会反问你 ===

反问题1：你为什么不用 LangChain 来做？
参考回答：我刻意不用框架，用 Python 标准库从头实现，可以深入理解
RAG 每个环节的原理。LangChain 封装太深，对学习不利。

反问题2：你的向量检索为什么用关键词而不是语义检索？
参考回答：为了避免依赖额外的 Embedding API，我设计了关键词检索 +
LLM 生成的解耦架构。在实际应用中展示效果很好，等接入 Embedding
API 后可以升级为混合检索。

反问题3：这个项目还有什么可以改进的？
参考回答：三个方向——1) 接入 Embedding API 做语义检索；
2) 增加混合检索和重排序；3) 支持 PDF 等更多文档格式。
